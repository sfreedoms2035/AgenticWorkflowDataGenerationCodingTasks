# DOCUMENTATION & ARCHITECTURE EXPLAINER

## Code Logic: High-Level Pipeline

This Rust framework encodes the abstract requirements of the EU AI Act (specifically Articles 10, 12, and 15) into core type concepts and traits, tailored for a high-performance ADAS environment where latency impacts must be minimized (< 1ms per loop cycle). The architecture prioritizes compile-time safety and formal interface contracts.

### Article 12: Record Keeping (Async Logging)
- **Problem**: Art. 12 mandates comprehensive, built-in logging (traceability) during the AI system's lifecycle. However, synchronous I/O in the main 30Hz processing loop is impossible due to performance constraints.
- **Solution**: The `ComplianceLogManager` implements a classic concurrent design. The ADAS loop receives a non-blocking `LogEnqueuer` containing a `mpsc::SyncSender`. Log events (`LogEvent`) are enqueued via a bounded channel. A dedicated background thread, `LogProcessor`, receives the events and performs buffered I/O to a CSV audit trail file. The manager handles startup and safe shutdown, ensuring all logs are flushed.

### Article 10: Data & Data Governance
- **Problem**: Art. 10 requires strict data quality, completeness, and bias prevention verification.
- **Solution**: Abstract requirements are formalized into traits (`CompletenessChecker<T>`, `BiasChecker<T>`) and a type-state pattern. `DataBatch<T, State>` uses `PhantomData` to track verification. Raw data (`DataBatch<T, Raw>`) is created but cannot be read until passed through `ComplianceValidator::validate()`. This method forces the execution of the traits (`DataQualityVerifier` and `BiasChecker`), which must return successfully before transitioning the state to `DataBatch<T, Verified>`. Verified data allows safe, secure reading.

### Article 15: Robustness and Accuracy
- **Problem**: Art. 15 requires "high levels of robustness, accuracy and cybersecurity." SOTIF scenarios necessitate mathematical verification of sensitivity against perturbations.
- **Solution**: The `RobustnessVerifier<T, In, Out>` trait provides a mathematical interface. It formalizes sensitivity analysis, accepting parameters like `epsilon` (allowed input perturbation range) to calculate `output_deviation_max` and `average_sensitivity_score`, derived from formal definitions like $S = \max_{\|\delta\| < \epsilon} \|f(x+\delta) - f(x)\|$.
